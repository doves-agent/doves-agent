任务队列

白鸽系统采用拉取（Pull）模式的任务队列设计。Server 只负责入队，Doves 自主轮询认领任务。Server 永远不推送。

核心设计：Server 写入任务 → Doves 主动拉取 → 原子认领 → 执行 → 回写结果。全程无推送。

状态 说明 触发条件
等待中 任务已创建，等待预处理 CLI/IM 提交任务
工具筛选中 系统正在为任务匹配可用工具 自动进入
已就绪 任务可被认领执行 工具筛选完成
执行中 已被鸽子认领，正在执行 原子认领成功

失败 执行失败（可重试） 执行异常或超时
已取消 用户主动取消 CLI/IM 发起取消
已终止 系统强制终止（不可重试） 监工强制中断
等待子任务 等待子任务完成后继续 鸽子创建子任务
阻塞中 等待外部条件满足 依赖未解决

原子任务认领

使用 MongoDB findOneAndUpdate 实现原子认领，确保同一任务不会被多只鸽子重复认领：

// 原子认领操作
db.tasks.findOneAndUpdate(
  {
    status: "已就绪",
    executor: null,                    // 未被认领
    capabilities: { $in: doveCaps },   // 能力匹配
    // 机器亲和检查（如有）
  },
  {
    $set: {
      status: "执行中",
      executor: doveId,
      claimedAt: new Date()
    }
  },
  { returnDocument: "after" }
)

认领前置检查

检查项 条件 未通过处理
鸽子状态 未被禁用 跳过本轮轮询
信誉分 >= 30 不允许认领任务
饲料余额 >= 1 不允许认领任务

机器亲和

标记 机器亲和: true 的任务只会被分配到匹配的机器上运行。用于需要特定硬件资源（GPU、大内存等）的任务。

轮询策略

轮询频率自适应：

活跃状态：1 秒/次
  │
  │ 连续 10 轮无任务
  ▼
空闲状态：3 秒/次
  │
  │ 遇到错误
  ▼
退避状态：5s × 2^n（最大 60s）+ 随机抖动 0~50%

示例退避序列：
  第 1 次错误：5s + jitter
  第 2 次错误：10s + jitter
  第 3 次错误：20s + jitter
  第 4 次错误：40s + jitter
  第 5 次错误：60s + jitter（达到上限）
  第 N 次错误：60s + jitter

成功认领任务 → 重置为活跃状态（1s）

子任务与依赖

鸽子在执行过程中可创建子任务，形成任务树：

主任务（执行中）
  ├── 子任务 A（已完成）
  ├── 子任务 B（执行中）
  │     └── 子子任务 B1（已就绪）
  └── 子任务 C（等待中）
      └── 依赖：子任务 B 完成后才能开始

主任务状态 → "等待子任务"（所有子任务完成后自动恢复）

检查点（Checkpoint）

支持执行过程中保存检查点，崩溃恢复后从检查点继续：

保存时机

每完成一轮工具调用后自动保存当前消息历史和执行状态。

恢复流程

鸽子重启后检测到未完成任务 → 加载检查点 → 从中断处继续执行。

重试机制

参数 值 说明
最大重试次数 由任务配置决定 超过后标记为"已终止"
重试间隔 指数退避 每次失败后间隔加倍
重试条件 状态为"失败" "已终止"不可重试

任务归档

归档策略：
  触发时间：每天凌晨 3:00（cron 调度）
  归档条件：已完成 > 48 小时的任务
  操作：从主集合移动到备份集合
  周期：24 小时

归档不影响用户查询历史任务的能力，系统会自动查询备份集合。

Change Stream 监控

Server 通过 MongoDB Change Stream 实时监控任务状态变更，用于：

  • 触发事件调度器的 data_change 类型事件
  • 实时通知用户任务完成（通过 IM 通道）
  • 更新统计指标
  • 触发后续流程（如子任务完成后恢复父任务）
