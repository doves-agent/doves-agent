架构总览

白鸽系统由三个核心组件构成，通过 Noise NX 加密 TCP 协议实现全链路安全通信。

系统组件

CLI

用户入口。提交任务、查询结果、管理鸽子。通过 DoveClient 与 Server 通信，同账号可直连 Doves。

Server

纯数据网关。负责认证、数据代理、任务入队。唯一端口 3003，不调用 LLM，不推送任务。

Doves

执行引擎。自主轮询拉取任务，调用 LLM 执行，运行扩展技能，回写结果。

架构拓扑

CLI ══加密TCP══▶ Server ◀══加密TCP══ Doves
 │                 │                    ▲
 │                 ▼                    │
 │         MongoDB / OSS / 快照 / 向量  │
 │         (仅 Server 有权直接访问)      │
 │                                      │
 └══════加密TCP（同账号直连）════════════─┘

三条通信线路

链路 用途 条件 协议
CLI → Server 提交任务、查询结果、用户管理 始终可用 Noise NX 加密 TCP
Doves → Server 轮询拉取任务、提交结果、心跳 始终可用 Noise NX 加密 TCP
CLI → Doves 管理鸽子、查看运行状态、扩展服务直调 同账号/同机器 Noise NX 加密 TCP

协议统一：三条线全部是 Noise NX 加密 TCP，没有第二种协议。无明文 HTTP 端口、无降级、无例外。

依赖与通信方向

允许 禁止
[加密] CLI → Server [明文] 任何组件间 HTTP 通信
[加密] Doves → Server [推送] Server → CLI
[加密] CLI → Doves（同账号直连） [推送] Server → Doves
[双向] IM ↔ Server [引用] Server → Doves 代码
[调用] Server → LLM
[调用] CLI → LLM
[暴露] 明文 HTTP 端口

设计原则

全链路加密

所有组件间通信统一 Noise NX 加密 TCP，无明文、无降级、无例外。

Server 不感知 Doves 逻辑

Server 不 import Doves 代码，不了解任务如何执行，只负责入队和存储结果。

拉取模式

Server 永远不主动推送任务。Doves 自主轮询拉取，保证执行端完全自治。

极致 KISS

总存在一个足够简单的实现恰好能满足所有场景。结构纯粹，场景自然对得上。
